## 6. 当前数据层机制

## 6.1 总体策略

H5 会员端当前采用：

- 后端优先请求
- 后端不可用时本地 mock / localStorage fallback

即：

1. 先请求 `/api/h5/...`
2. 如果后端未就绪、401、服务不可用、或允许 fallback 的情况下
3. 切到本地 mock 数据

这套策略是当前 H5 能持续开发的关键，不要轻易删掉。

## 6.2 本地存储 key

`frontend/src/services/h5Member.ts` 当前使用：

- `frontend.h5.member-accounts.v1`
- `frontend.h5.member-states.v1`
- `frontend.h5.member-session.v1`

作用分别是：

- 会员账户列表
- 会员状态数据
- 当前登录 session

## 6.3 当前 mock 支持的能力

本地 mock 当前已经支持：

- 注册
- 登录
- 登出
- 首页 dashboard
- 任务包列表 / 详情
- 任务购买流程
- 双余额
- 充值
- 提现
- 任务余额转系统余额
- 提现记录
- 订单列表
- 消息列表
- 工单列表 / 详情 / 回复 / 新建
- 推广页基础数据
- 碎片系统基础数据
- WhatsApp 绑定状态
- 认证申请与状态
- 个人设置页：头像、手机号、密码修改

## 6.4 个人设置 mock 说明

新增的以下两个方法会写本地会员数据：

- `updateMemberProfile(...)`
- `updateMemberPassword(...)`

效果：

- 修改手机号后会更新当前 session 和 member account
- 上传头像后会把 base64 Data URL 存到本地 mock 账户
- 修改密码后会更新本地账户密码，下次登录生效

注意：

- 这不是正式后端持久化能力
- 这是为了保证前端原型“真可点、真可改、真可继续测”

## 9. 当前代码结构上的明显问题

这是给新会话的真实提醒。

## 9.1 `H5App.tsx` 太大

当前 H5 功能几乎都在一个文件内。

风险：

- 继续叠需求会越来越难维护
- 局部改动容易误伤别的页面
- 状态变量过多

建议后续优先做的技术整理：

- 保持现有功能不变
- 逐步把页面拆到 `frontend/src/pages/h5-member/`

可优先拆：

- `ProfilePage`
- `SettingsPage`
- `RechargePage`
- `WithdrawPage`
- `PromotionPage`

## 9.2 `styles.css` 混合度高

它不只是 H5 会员端样式。

风险：

- 后续改 H5 时容易撞到别的模块
- 有一些过时的 H5 选择器残留

建议：

- 后续逐步把 H5 样式抽到独立文件
- 或至少把 H5 样式集中到一段清晰区域

## 9.3 文案硬编码仍然很多

当前 `H5App.tsx` 里仍有很多中文文案直接写在 JSX 内。

这会影响后续：

- 多语言切换
- 统一文案维护

后续建议：

- 先抽一个 H5 文案字典
- 再做语言切换

## 9.4 后端能力仍大量缺失

不要误以为当前前端已经进入正式联调阶段。

当前大量能力仍是 mock：

- 任务包完整后端闭环
- 双余额账本正式持久化
- 充值 / 提现正式流程
- 推广归因统计
- 碎片闭环正式后端
- WhatsApp 正式绑定

详见：

- `docs/h5-backend-gap-report.md`

## 10. 当前最应该继续做的前端工作

如果新会话接着做前端，我建议按下面顺序。

## P1：组件拆分，不改业务结果

优先做：

- 拆 `H5App.tsx`
- 把“我的 / 设置 / 充值 / 提现 / 推广”先拆出去

原因：

- 这些页面近期改动最多
- 再继续堆功能风险很高

## P2：整理 H5 文案与 i18n 入口

优先做：

- 抽出 H5 文案常量
- 明确中文文案 key
- 给后续英文切换留接口

原因：

- 当前已经明确后期主流要英文
- 现在继续堆中文硬编码成本会更高

## P3：清理重复入口与信息层级
