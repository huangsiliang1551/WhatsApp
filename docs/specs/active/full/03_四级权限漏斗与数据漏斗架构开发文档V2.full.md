> 适用仓库：`https://github.com/huangsiliang1551/WhatsApp`  
> 目标读者：Codex 开发线程、后端、前端、测试、产品  
> 重要性：P0 / S 级  
> 版本：V2  
> 核心目标：将当前较分散的权限体系，重构为“超级管理员 → 代理商 → 主管 → 成员”的四级权限漏斗，并同步建立“数据漏斗、客户归属漏斗、会话转接漏斗、报表归属快照”四套规则，避免财务、客户、会话、任务、报表各写各的权限逻辑。

## 0. 最终架构结论

本项目权限体系统一为四级：

```text
超级管理员 Super Admin
  ↓ 授权权限 + 数据范围
代理商 Agency
  ↓ 授权权限 + 数据范围
主管 Supervisor
  ↓ 授权权限 + 数据范围
成员 Staff Member
```

核心原则：

```text
权限只能向下授权，不能越级放大。
数据只能向下看到，不能越级穿透。
下级拥有的权限 <= 上级授予的权限。
下级可见的数据 <= 上级授予的数据范围。
```

因此，系统必须同时有两条漏斗：

```text
权限漏斗 Permission Funnel
数据漏斗 Data Scope Funnel
```

权限漏斗决定：

```text
能不能看财务
能不能审批提现
能不能追加任务商品
能不能转接会话
能不能导出报表
```

数据漏斗决定：

```text
看谁的数据
看哪个客户
看哪个会话
看哪个成员名下的充值提现
看哪个团队的报表
```

## 1. 当前源码现状与问题

### 1.1 当前已有基础

当前源码已经有 `RequestActor`、`ActorRole`、`has_permission()`、`require_permission()`、`require_account_access()` 等基础，`super_admin` 默认拥有全部权限，普通用户通过 `resolved_permissions` 判断权限。当前 `ActorRole` 包含 `super_admin/operator/reviewer/support_agent/finance/risk_control/readonly/agent/agent_member` 等角色。

当前 `app/api/deps.py` 里已经有：

```text
get_request_actor()
get_strict_request_actor()
require_permission(permission_code)
```

并且在生产环境不会信任 `X-Actor-*` header，只在 development/test 场景允许 header actor。JWT 中的 agent / agent_member 会动态解析 RolePermission。

当前 `app/core/permission_defs.py` 已经有集中权限定义 `PERMISSION_DEFINITIONS`，并包含 code / module / label / description / super_admin_only 等字段。

### 1.2 当前主要问题

当前问题不是“没有权限”，而是：

```text
1. 权限角色体系不是你想要的四级漏斗。
2. ActorRole、user_type、permission_role、RolePermission、site role 多套角色并存。
3. require_permission 只管权限码，不自动管数据范围。
4. account_ids 只能解决账号范围，不能解决主管和成员之间的数据层级。
5. 前端菜单没有统一 requiredPermissions / dataScope 规则。
6. 会话转接、客户归属、财务报表、任务监控还缺统一数据归属规则。
7. 报表容易因为客户归属调整而出现历史数据口径变化。
```

因此，本次重构不是简单加权限码，而是要建立：

```text
权限授权链
数据归属链
团队层级
客户归属
会话分配
报表快照
敏感字段权限
审计日志
```

## 2. 四级组织与角色模型

### 2.1 四级身份

统一身份层级：

| 层级 | 英文 | 说明 |
|---|---|---|
| 超级管理员 | `super_admin` | 平台最高权限，拥有全部权限和全部数据 |
| 代理商 | `agency` / `agency_admin` | 代理商主体或代理商管理员，拥有超级管理员授权的权限和数据 |
| 主管 | `supervisor` | 代理商下的团队负责人，拥有代理商授权的权限和团队数据 |
| 成员 | `staff_member` | 主管下的客服/运营/财务/任务操作员，拥有主管授权的权限和自己的数据 |

注意：

```text
“成员”在本文指后台成员/客服/员工，不是 H5 客户。
H5 客户统一称为 customer / member_profile / h5_user。
```

### 2.2 推荐实体

建议在代码中清晰区分：

```text
Agency              代理商主体
StaffUser           后台工作人员，可以是代理商管理员、主管、成员
StaffTeam           团队
StaffTeamAssignment 成员归属哪个主管/团队
CustomerOwnership   客户归属哪个成员/主管/代理商
ConversationAssignment 会话当前分配给谁
```

如果当前项目已有 `Agent` / `AgencyMember`，可以先复用，不必立刻新建 StaffUser 表，但概念上必须统一。

### 2.3 角色不是数据范围

不要把角色和数据范围混为一谈。

例如：

```text
角色：主管
权限：finance.withdrawal.view
数据范围：只能看 Team A 的客户提现
```

同样是主管，不同主管的数据范围不同：

```text
主管A 管理 3 个客服 → 只能看这 3 个客服及其客户
主管B 管理 5 个客服 → 只能看这 5 个客服及其客户
```

因此必须独立存：

```text
role_key
permission_grants
data_scope_grants
team_assignments
customer_ownership_assignments
```

## 3. 权限漏斗设计

### 3.1 权限向下授权规则

权限继承规则：

```text
超级管理员拥有全部权限。
代理商只能获得超级管理员授予的权限。
主管只能获得代理商授予的权限。
成员只能获得主管授予的权限。
```

公式：

```text
成员有效权限
= 成员被授予权限
∩ 主管有效权限
∩ 代理商有效权限
∩ 平台允许权限
```

主管有效权限：

```text
主管有效权限
= 主管被授予权限
∩ 代理商有效权限
```

代理商有效权限：

```text
代理商有效权限
= 超级管理员授予代理商的权限
```

超级管理员：

```text
super_admin = all permissions
```

### 3.2 权限授权必须区分“拥有”和“可继续下发”

某个角色拥有权限，不一定能继续给下级。

例如：

```text
代理商可以查看财务 finance.report.view
但不一定能把 finance.report.view 授权给主管。
```

因此权限授予表要有：

```text
permission_code
granted_to_subject
granted_by_subject
can_delegate
max_scope
status
```

字段建议：

```python
class PermissionGrant(Base):
    __tablename__ = "permission_grants"

    id = Column(String(64), primary_key=True)

    grantor_subject_type = Column(String(32), nullable=False)
    grantor_subject_id = Column(String(64), nullable=False)

    grantee_subject_type = Column(String(32), nullable=False)
    grantee_subject_id = Column(String(64), nullable=False)

    permission_code = Column(String(128), nullable=False, index=True)

    can_delegate = Column(Boolean, nullable=False, default=False)

    scope_type = Column(String(32), nullable=False, default="inherit")
    # global / agency / team / own / inherit

    status = Column(String(32), nullable=False, default="active")
    # active / revoked / expired

    expires_at = Column(DateTime(timezone=True), nullable=True)

    created_by = Column(String(64), nullable=False)
    created_at = Column(DateTime(timezone=True), nullable=False)
    revoked_by = Column(String(64), nullable=True)
    revoked_at = Column(DateTime(timezone=True), nullable=True)
```

### 3.3 授权校验

当代理商给主管授权时：

```text
代理商自己必须拥有该权限
代理商必须拥有该权限的 can_delegate=true
主管授权范围不能超过代理商范围
```

当主管给成员授权时：

```text
主管自己必须拥有该权限
主管必须拥有该权限的 can_delegate=true
成员授权范围不能超过主管范围
```

如果上级权限被撤销：

```text
下级保留授权记录，但 effective=false
前端显示“因上级权限撤销而失效”
```

不要直接删除下级权限记录，否则审计和历史无法追踪。

## 4. 数据漏斗设计

### 4.1 数据范围规则

数据范围同样向下收窄：

```text
超级管理员：全部数据
代理商：自己的代理商数据
主管：自己管理的团队/成员/客户数据
成员：自己的客户/会话/任务/订单数据
```

公式：

```text
资源是否可见
= 资源属于上级授权数据范围
AND 资源属于当前角色数据范围
AND 当前角色拥有对应权限
```

### 4.2 数据范围实体

建议新增：

```python
class DataScopeGrant(Base):
    __tablename__ = "data_scope_grants"

    id = Column(String(64), primary_key=True)

    subject_type = Column(String(32), nullable=False)
    subject_id = Column(String(64), nullable=False)

    scope_type = Column(String(32), nullable=False)
    # global / agency / account / site / team / staff / customer

    scope_id = Column(String(64), nullable=True, index=True)

    granted_by_subject_type = Column(String(32), nullable=False)
    granted_by_subject_id = Column(String(64), nullable=False)

    status = Column(String(32), nullable=False, default="active")

    created_at = Column(DateTime(timezone=True), nullable=False)
    revoked_at = Column(DateTime(timezone=True), nullable=True)
```

### 4.3 各层默认数据范围

| 角色 | 默认数据范围 |
|---|---|
| 超级管理员 | `global` |
| 代理商 | `agency:{agency_id}`，以及代理商名下 account/site |
| 主管 | `team:{team_id}` 或 `supervisor:{supervisor_id}` |
| 成员 | `staff:{staff_id}` |
| H5客户 | 只访问自己的 H5 数据，不进入后台权限体系 |

### 4.4 数据访问不要依赖创建人

你问到：

> 主管看到的成员数据，是看成员在哪个主管创建的，还是代理商后台分配？

建议答案：

```text
以当前团队/主管分配为准，不以创建人为准。
创建人只用于审计。
```

原因：

```text
1. 成员可能由代理商创建，再分配给主管。
2. 成员可能从主管A调到主管B。
3. 客户可能从成员A转给成员B。
4. 如果按 created_by 判断，组织调整后数据会乱。
```

因此必须有两个概念：

```text
created_by：谁创建的，只用于审计
current_supervisor_id / current_team_id：当前归属，用于权限和数据范围
```

### 4.5 成员归属可调整

必须支持：

```text
代理商把成员从主管A调整到主管B
主管把成员移出自己团队（如果有权限）
超级管理员跨代理商调整（一般不建议，但可提供超级操作）
```

调整时要选择数据迁移策略：

```text
仅调整成员以后产生的数据
同时迁移该成员当前客户
同时迁移该成员当前会话
同时迁移该成员未完成任务
```

默认建议：

```text
仅调整成员归属，不自动迁移客户和历史数据。
客户/会话/任务归属需要单独转移。
```

否则成员调岗会导致历史报表口径混乱。

## 7. 财务板块数据漏斗

### 7.1 财务数据的两种口径

财务非常特殊，必须分清：

```text
当前归属口径 current ownership
历史快照口径 snapshot ownership
```

#### 当前归属口径

用于：

```text
提现审核列表
当前客户财务概览
客服当前跟进
```

看的是客户现在归属于谁。

#### 历史快照口径

用于：

```text
财务日报
客服业绩
主管业绩
代理商历史报表
充值提现统计
任务奖励统计
```

看的是交易发生时客户归属于谁。

### 7.2 所有资金流水必须保存归属快照

以下表或流水必须保存：

```text
owner_staff_id_snapshot
supervisor_id_snapshot
team_id_snapshot
agency_id_snapshot
site_id_snapshot
account_id_snapshot
```

适用：

```text
WalletLedgerEntry
RechargeRecord
WithdrawalRequest
WalletBonusGrantRecord
RechargeRepairOrder
TaskPackageInstance reward ledger
TaskManualAddItemLog
PaymentReconciliationItem
```

### 7.3 财务可见性

#### 超级管理员

```text
全平台全部财务
```

#### 代理商

```text
agency_id = 当前代理商
```

#### 主管

默认：

```text
当前审核类列表：current_supervisor_id = 当前主管
历史报表：supervisor_id_snapshot = 当前主管
```

可以加切换：

```text
报表口径：
- 按交易发生时归属
- 按当前客户归属
```

默认建议：

```text
报表使用交易发生时归属。
```

#### 成员

默认：

```text
当前客户财务：current_owner_staff_id = 当前成员
历史报表：owner_staff_id_snapshot = 当前成员
```

### 7.4 财务例子

假设：

```text
代理商 A
主管 S1 管理成员 M1/M2/M3
主管 S2 管理成员 M4/M5/M6/M7/M8
```

财务可见：

```text
超级管理员：A、B、C 所有代理商财务
代理商A：A 下全部财务
主管S1：M1/M2/M3 的客户财务
主管S2：M4/M5/M6/M7/M8 的客户财务
成员M1：自己客户财务
```

如果 M1 从 S1 调到 S2：

```text
调岗后的新交易 snapshot 到 S2。
调岗前的历史业绩仍属于 S1，除非做历史归属迁移。
```

这可以避免主管之间因为成员调岗导致历史报表互相“抢数据”。

## 8. 会话板块与AI转接

### 8.1 会话为什么复杂

会话涉及：

```text
客户归属
当前客服
AI自动接待
临时转接
永久转移客户归属
团队队列
在线状态
自动分配规则
```

必须区分两个动作：

```text
会话临时转接 conversation handover
客户永久归属转移 customer ownership transfer
```

不要把这两个混为一个操作。

### 8.2 ConversationAssignment

```python
class ConversationAssignment(Base):
    __tablename__ = "conversation_assignments"

    id = Column(String(64), primary_key=True)

    conversation_id = Column(String(64), nullable=False, index=True)
    customer_id = Column(String(64), nullable=False, index=True)

    agency_id = Column(String(64), nullable=False, index=True)
    team_id = Column(String(64), nullable=True, index=True)
    supervisor_id = Column(String(64), nullable=True, index=True)

    assigned_staff_id = Column(String(64), nullable=True, index=True)
    assigned_queue_id = Column(String(64), nullable=True, index=True)

    assignment_type = Column(String(32), nullable=False)
    # customer_owner / ai_handover / manual_transfer / supervisor_assign / queue_assign

    is_temporary = Column(Boolean, nullable=False, default=True)

    status = Column(String(32), nullable=False, default="active")
    # active / ended

    assigned_by = Column(String(64), nullable=True)
    assigned_at = Column(DateTime(timezone=True), nullable=False)
    ended_at = Column(DateTime(timezone=True), nullable=True)

    reason = Column(Text, nullable=True)
```

### 8.3 会话状态机

建议会话状态：

```text
ai_active              AI正常接待
waiting_human          AI遇到问题，等待人工
assigned_to_staff      已分配给客服
human_active           人工接待中
transferred            已转接
closed                 已关闭
```

### 8.4 普通客服可见性

成员只能看到：

```text
conversation.assigned_staff_id = 当前成员

```

成员只能回复：

```text
assigned_staff_id = 当前成员
```

如果只是客户归属自己，但会话当前被转给别人：

```text

```

### 8.5 主管可见性

主管可看到：

```text

OR supervisor_id = 当前主管
```


不能默认：

```text

```

```text

```

### 8.6 代理商可见性

```text

```

### 8.7 AI自动转接给谁

AI遇到问题时必须走明确规则，不要随机乱转。

建议转接优先级：

```text
1. 客户当前归属客服 current_owner_staff_id，如果在线且可接单
2. 客户当前主管团队的可用成员，按队列规则分配
3. 当前主管本人或主管指定兜底成员
4. 代理商级默认人工队列
5. 代理商管理员/总队列
6. 如果无人可接，进入 waiting_human 队列并告警
```

### 8.10 会话转接和客户归属转移的区别

#### 临时会话转接

```text
只改变 ConversationAssignment
不改变 CustomerOwnershipAssignment
适合：
- 客服临时离线
- AI转人工
- 咨询升级
```

#### 永久客户转移

```text
改变 CustomerOwnershipAssignment
未来客户会话、财务当前归属、任务监控都跟新归属
适合：
- 客户换客服
- 成员离职
- 团队调整
```

前端必须两个按钮分开：

```text
转接当前会话
转移客户归属
```

不要一个“转接”按钮同时改两套。

## 16. 禁止事项

1. 不要用 created_by 判断主管数据范围。
2. 不要把会话转接等同于客户归属转移。
3. 不要让主管看到其他主管成员数据。
4. 不要让成员看到团队报表，除非显式授权且数据范围仍限制为自己。
5. 不要用单纯 account_id 过滤替代团队/成员过滤。
6. 不要让代理商授权自己没有的权限。
7. 不要让主管授权自己没有的权限。
8. 不要因为成员调岗而自动改历史财务报表。
9. 不要让 AI handover 随机分配给任意客服。
10. 不要只在前端隐藏菜单，后端必须强制 scope 过滤。

## 17. 最终 Definition of Done

完成后必须满足：

1. 系统有清晰四级架构：超级管理员、代理商、主管、成员。
2. 权限从上到下漏斗式授权。
3. 数据从上到下漏斗式可见。
4. 代理商权限不能超过超级管理员授予。
5. 主管权限不能超过代理商授予。
6. 成员权限不能超过主管授予。
7. 主管只能看自己团队成员的数据。
8. 成员只能看自己的数据。
9. 财务、客户、会话、任务、报表都接入数据漏斗。
10. 成员归属以当前团队分配为准，不以创建人为准。
