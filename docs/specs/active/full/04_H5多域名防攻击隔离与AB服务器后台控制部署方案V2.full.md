## 0. 一句话目标

后端和核心服务部署在 A 服务器，N 多 H5 域名和静态站点部署在 B 服务器。管理员在 A 服务器后台配置 H5 域名、B 节点和安全策略后，后台可以自动完成：

```text
1. 检查 B 服务器连通性
2. 初始化 B 服务器环境
3. 安装/更新 H5 网关
4. 自动拉取 H5 前端包
5. 自动生成/续期证书
6. 自动生成 Nginx/Caddy 配置
7. 自动配置防火墙和 CDN 回源白名单
8. 自动验证域名解析
9. 自动检查端口占用
10. 自动 reload / rollback
11. 自动采集 B 服务器运行状态
12. 自动发现攻击并封禁单个 H5 域名
13. 自动反馈部署结果和错误日志到后台
```

最终效果：

```text
后台创建 H5 站点 + 绑定域名 + 选择 B 节点
  ↓
点击部署
  ↓
后台自动控制 B 服务器完成所有操作
  ↓
H5 域名可以直接访问
```

某个 H5 域名被攻击时：

```text
只封禁该域名
不影响 B 服务器上的其他 H5 域名
不影响 A 服务器后端业务
```

## 1. 当前源码基础和必须改造点

### 1.1 当前已有能力

当前源码已有：

```text
app/api/routes/h5_deploy.py
app/services/h5_deploy_service.py
app/services/domain_verification_service.py
app/core/rate_limit_middleware.py
frontend/nginx.conf
```

当前 `H5DeployService.generate_deploy_script()` 是“生成脚本给人执行”的模式，并且脚本会为每个站点生成一套：

```text
app container
frontend container
ports 8000 / 80
```

这不适合你的目标，因为你需要：

```text
A服务器跑后端
B服务器只跑 H5 网关和静态前端
B服务器承载 N 多 H5 域名
每个域名不单独占端口
后台自动控制 B服务器，不让人工复制脚本执行
```

因此需要把当前 H5 部署模式从：

```text
每站点生成 docker-compose.h5.yml 给人手动执行
```

改成：

```text
A后台创建部署任务
  ↓
A worker 通过 SSH 或 B Agent 自动执行脚本
  ↓
B服务器返回执行步骤、日志、健康状态
```

### 1.2 当前必须避免的问题

不要继续沿用当前 H5 deploy 的生产方式：

```text
1. 不要每个 H5 站点都起一个 app 容器。
2. 不要每个 H5 站点都抢 80/8000 端口。
3. 不要把 B服务器变成后端服务器。
4. 不要让 H5 域名可以访问后台 API。
5. 不要让 B服务器的操作结果只通过人工观察。
6. 不要让证书靠人工申请。
7. 不要让 B服务器安全配置靠人工记忆。
8. 不要让后台不知道 B 当前 nginx 配置、证书、端口和攻击状态。
```

## 2. 推荐控制架构：SSH Bootstrap + B Agent 常态控制

### 2.1 为什么不建议全靠 SSH

你希望后台能给 B 服务器 SSH 权限，然后后台自动执行。这个可以做，但如果所有日常操作都通过 SSH 直接执行，风险较高：

```text
1. A服务器一旦被入侵，攻击者可能直接控制 B服务器。
2. SSH私钥存储、命令权限、sudo权限都很敏感。
3. 长命令和脚本执行失败时，状态回传不够稳定。
4. 多个后台任务并发 SSH 容易冲突。
5. 日常配置同步更适合 agent pull。
```

### 2.2 推荐模式

采用双模式：

```text
模式A：SSH Bootstrap 模式
用于第一次初始化 B服务器、安装 agent、安装 nginx/certbot/firewall、创建目录、创建部署用户。

模式B：B Gateway Agent 常态模式
用于日常同步域名配置、部署H5包、证书续期、监控上报、攻击事件回传、执行白名单脚本。
```

这样既能满足你“后台最大可能操控 B服务器”的目标，又降低长期 SSH 风险。

### 2.3 最终控制链路

```text
A后台
  ↓ 创建部署/检查/封禁/证书/升级任务
A Worker
  ↓
如果B未初始化：SSH执行 bootstrap
如果B已初始化：调用 B Agent API / B Agent主动拉任务
  ↓
B Agent 执行白名单脚本
  ↓
B Agent 回传 step log / status / metrics
  ↓
A后台展示执行结果
```

### 2.4 SSH 使用边界

SSH 只用于：

```text
1. 首次安装 B Gateway Agent
2. 修复 Agent 失联
3. 紧急回滚
4. 人工授权的维护动作
```

常规动作不建议直接 SSH：

```text
发布H5前端包
新增域名
reload nginx
自动续证书
封禁域名
采集指标
```

这些由 B Agent 执行。

## 3. A/B服务器职责

### 3.1 A服务器职责

A服务器运行：

```text
Postgres
Redis
FastAPI app
Worker / Scheduler
后台前端
Prometheus / Grafana / Alertmanager
支付、提现、任务、财务、WhatsApp、AI 等核心服务
```

A服务器负责：

```text
1. B节点注册和授权
2. SSH凭据管理
3. Agent Token管理
4. H5站点和域名配置
5. 生成网关配置
6. 生成部署任务
7. 接收B节点心跳
8. 接收B节点执行结果
9. 监控B节点状态
10. 触发封禁/解封/证书/更新/回滚
```

### 3.2 B服务器职责

B服务器运行：

```text
Nginx / OpenResty / Caddy
h5-gateway-agent
H5静态前端文件
Certbot / acme client
防火墙规则
日志采集
```

B服务器负责：

```text
1. 承载所有 H5 域名
2. 按 Host 识别站点
3. 静态资源本地返回
4. /api/h5/* 反代到 A
5. 禁止 /api/admin、/api/finance 等后台接口
6. 域名级限流
7. 域名级封禁
8. 证书自动申请/续期
9. 上报 QPS、错误率、证书状态、Nginx状态
10. 上报攻击事件
```

### 3.3 B服务器不应该做的事

```text
1. 不跑 A 后端 app。
2. 不连接数据库。
3. 不保存支付/提现/任务核心数据。
4. 不保存 Meta/支付/AI密钥。
5. 不暴露后台管理接口。
6. 不给每个H5站点单独占端口。
```

class H5GatewayNode(Base, TimestampMixin):

## 13. 端口不重复设计

### 13.1 生产架构避免端口冲突

N多H5域名不应该每个域名占一个端口。全部使用：

```text
80
443
```

通过：

```text
Host
server_name
map
```

区分站点。

因此不会出现：

```text
h5a 占用 8081
h5b 占用 8082
h5c 占用 8083
```

### 13.2 B初始化时检查端口

`bootstrap_b_node.sh` 必须检查：

```text
80 是否被其他服务占用
443 是否被其他服务占用
```

如果被占用：

```text
返回 PORT_CONFLICT
后台显示哪个进程占用
不继续覆盖
```

命令：

```bash
ss -ltnp | grep ':80 '
ss -ltnp | grep ':443 '
```

### 13.3 Agent端口

如果 B Agent 采用 pull 模式：

```text
不需要开放agent公网端口
```

Agent 主动连接 A。

如果采用 push 模式：

```text
agent 监听 127.0.0.1 或内网IP
只允许 A服务器访问
```

推荐：

```text
pull mode
```

## 14. 防火墙与安全配置

### 14.1 B服务器安全组

建议：

```text
入站 80/443：只允许 CDN/WAF IP 段
入站 22：只允许 A服务器IP/VPN/办公IP
入站其他：拒绝
出站：允许访问 A服务器、DNS、Let's Encrypt、CDN API、系统更新源
```

如果没有 CDN：

```text
80/443 对公网开放，但必须启用Nginx限流。
这只能防应用层，不能防带宽打满。
```

### 14.2 Origin Secret Header

如果走 CDN，在 CDN 回源中添加：

```text
X-Origin-Verify: strong-random-secret
```

B Nginx 校验该 header。

没有该 header：

```text
403
```

### 14.3 Authenticated Origin Pull / mTLS

如果 CDN 支持，建议启用：

```text
CDN 到 B 的 mTLS
```

这样即使攻击者知道 B IP，也无法模拟 CDN 回源。

### 14.4 A服务器安全组

A服务器：

```text
H5 API 只允许 B服务器内网/VPN IP
Admin 只允许后台域名/CDN/VPN
Webhook 允许 Meta，但必须验签
支付回调允许通道IP，但必须验签
DB/Redis 不开放公网
```

## 23. 禁止事项

1. 不要让后台执行任意SSH命令。
2. 不要把SSH私钥明文存库。
3. 不要用 root 账号作为常规部署用户。
4. 不要让 sudoers 配置 `NOPASSWD: ALL`。
5. 不要每个H5站点单独占端口。
6. 不要让B服务器跑A后端app。
7. 不要让H5域名访问后台管理API。
8. 不要在证书失败时覆盖现有可用证书。
9. 不要在nginx -t失败时reload。
10. 不要在配置失败时覆盖旧配置。
11. 不要让release部署失败后current指向坏版本。
12. 不要让B Agent失联时后台误判部署成功。
13. 不要把DNS自动修改作为默认能力，除非配置DNS provider token。
14. 不要让CDN IP白名单过期无人提醒。
15. 不要让blocked域名请求继续打到A服务器。

## 24. Definition of Done

完成后必须满足：

1. 后台能新增B服务器节点。
2. 后台能保存加密SSH凭据。
3. 后台能测试SSH连接。
4. 后台能一键初始化B服务器。
5. B服务器能自动安装网关、agent、证书工具、防火墙基础配置。
6. B Agent 能心跳上报。
7. 后台能看到B服务器CPU、内存、磁盘、nginx、agent、证书状态。
8. H5前端包能由B服务器自动拉取并校验sha256。
9. H5前端包能原子切换和回滚。
10. 后台能为H5域名自动检查DNS。
11. 后台能触发证书申请和续期。
12. 证书失败不会影响旧证书。
13. 后台能生成并下发Nginx域名配置。
14. Nginx配置必须通过 `nginx -t` 才能reload。
15. 一个B服务器可以承载多个H5域名，不产生端口冲突。
16. B服务器只开放80/443，H5站点通过Host区分。
17. H5域名只能访问H5必要API。
18. H5域名不能访问后台管理API。
19. 后台可以手动封禁/解封单个H5域名。
20. 自动检测攻击后可以封禁单个H5域名。
21. 被封域名不再把请求转发到A服务器。
22. 其他H5域名不受影响。
23. 所有B服务器操作都有Job、Step、日志、状态、审计。
24. 所有脚本支持dry-run和JSON输出。
25. 部署、证书、防火墙、H5发布、域名封禁均有测试覆盖。
