# H5 会员端 v6 页面级线框与文案整改稿

## 1. 文档目标

本稿用于把 H5 会员端从“功能原型”推进到“更适合上线验证的产品原型”。

本轮重点不再增加功能，而是聚焦以下 3 件事：

1. 让用户更快进入主链路
2. 让资金和任务规则更容易理解
3. 让失败、超时、提现等高投诉场景更可预期

主链路固定为：

1. 注册 / 登录
2. 进入首页
3. 领取任务包
4. 充值系统余额
5. 完成任务包内购买
6. 获得任务余额
7. 划转至系统余额
8. 达到门槛后提现
9. 查看提现状态

后续所有页面整改，都应优先服务这条链路。

## 2. 首页

### 页面目标

用户进入首页后，3 秒内知道“现在最该做什么”。

### 页面结构

#### 2.1 顶部信息栏

展示内容：

- 站点名
- 脱敏账号 ID
- 未读消息数

要求：

- 保持极简，不放欢迎语和大段说明
- 不放营销 Banner

#### 2.2 主动作卡片

首页首屏只允许有一个主 CTA，根据用户状态动态变化。

状态 A：有待领取任务包

- 标题：`您有新的任务包可领取`
- 副文案：`领取后将开始计时，请在规定时间内完成`
- 主按钮：`立即领取`

状态 B：有进行中任务包

- 标题：`您有进行中的任务`
- 副文案：`继续完成任务可获得任务奖励`
- 主按钮：`继续任务`

状态 C：系统余额不足

- 标题：`当前系统余额不足`
- 副文案：`请先充值系统余额后再继续任务`
- 主按钮：`去充值`

状态 D：已满足提现条件

- 标题：`当前已满足提现条件`
- 副文案：`可申请提现并查看最新状态`
- 主按钮：`去提现`

要求：

- 首页首屏不能并列放多个同级主按钮
- 碎片、推广、排行榜都不能抢首屏主视觉

#### 2.3 钱包摘要卡

展示字段：

- 系统余额
- 任务余额
- 当前提现状态
- 距提现门槛差额

建议文案：

- 系统余额：`用于购买商品，也可用于提现`
- 任务余额：`任务完成后到账，需先划转`
- 当前提现状态：
  - 满足门槛：`已满足提现门槛`
  - 未满足门槛：`距离提现门槛还差 {金额}`

底部短按钮：

- `充值`
- `划转`
- `提现`

要求：

- 钱包摘要卡应出现在首页前 2 屏内
- 用户不进入钱包页，也能知道当前钱的状态

#### 2.4 任务摘要区

展示：

- 待领取任务包数量
- 进行中任务包数量
- 即将过期任务包数量

建议标签：

- `待领取`
- `进行中`
- `即将超时`

点击后分别跳转对应任务筛选页。

#### 2.5 快捷入口区

保留入口：

- 任务中心
- 充值 / 提现
- 联系客服
- 推广任务
- 碎片背包
- 绑定 WhatsApp

要求：

- 快捷入口是工具区，不是主引导区
- 视觉优先级低于主动作卡片和钱包摘要

#### 2.6 重要通知区

只展示高优先级消息：

- 任务即将过期
- 提现状态更新
- 工单回复
- 购买失败提醒

文案示例：

- `您有 1 个任务包将在 2 小时内到期`
- `您的提现申请已进入审核中`
- `您的工单已收到平台回复`

#### 2.7 提现排行榜

展示字段：

- 脱敏账号 ID
- 累计提现金额

辅助文案：

- `仅展示有效提现记录`

要求：

- 排行榜放首页靠后区域
- 不做夸张轮播和过强运营化视觉

## 3. 任务页

### 页面目标

让用户快速找到当前该做的任务，并顺利继续执行。

### 页面结构

#### 3.1 顶部筛选

默认主筛选：

- `进行中`
- `待领取`
- `已完成`

次级筛选入口：

- 新手
- 成长
- 推广
- 已失效

要求：

- 进入任务页默认展示“进行中”
- 有进行中任务时优先展示进行中任务包

#### 3.2 任务包列表卡片

每张卡片主信息只保留：

- 任务包名称
- 类型标签
- 倒计时
- 当前进度
- 总佣金
- 主按钮

副信息：

- 商品数量
- 奖励比例

卡片文案示例：

- `新手任务包 A`
- `共 8 件商品`
- `奖励比例 12%`

状态文案：

- `待领取`
- `进行中`
- `已完成`
- `已失效`

按钮文案：

- `立即领取`
- `继续任务`
- `查看结果`

要求：

- 卡片主次信息必须分层
- 首眼先看到“值不值得做、还剩多久、做到哪了”

#### 3.3 领取确认弹层

点击领取后必须弹出规则确认层。

标题：

- `确认领取任务包`

正文：

- `领取后将立即开始计时`
- `请在 24 小时内完成全部任务`
- `任务完成后奖励将进入任务余额`
- `超时后任务包自动失效`

按钮：

- `取消`
- `确认领取`

## 4. 任务详情页

### 页面目标

让用户知道自己还差多少完成，以及接下来点哪个商品。

### 页面结构

#### 4.1 顶部任务摘要条

展示：

- 已完成 `X / Y`
- 剩余商品数
- 已累计佣金
- 完成后预计到账任务余额
- 剩余倒计时

文案示例：

- `已完成 3 / 8`
- `剩余 5 件商品`
- `已累计佣金 $12.50`
- `完成后预计到账任务余额 $38.00`
- `剩余时间 06:12:44`

#### 4.2 商品列表

每个商品展示：

- 商品图
- 商品名
- 商品价格
- 当前累计佣金
- 状态
- 按钮

按钮状态：

- `购买`
- `处理中`
- `已完成`
- `失败重试`

#### 4.3 购买流程浮层

点击购买后在当前页内展示浮层，不跳页面。

流程文案：

1. `正在创建订单`
2. `正在付款`
3. `付款成功` / `付款失败`
4. `任务完成` / `任务失败`

要求：

- 总时长控制在 3 秒内
- 成功后要明显反馈进度增长
- 失败后不能停在纯状态，要给下一步动作

#### 4.4 失败补救文案

余额不足：

- 标题：`余额不足，暂时无法购买`
- 按钮：`去充值`

支付失败：

- 标题：`支付未成功，请重新尝试`
- 按钮：`重新支付`

系统异常：

- 标题：`本次操作未完成，请稍后重试`
- 按钮：`重新尝试`
- 次按钮：`联系客服`

任务过期：

- 标题：`任务包已超过完成时间`
- 按钮：`返回任务中心`

## 5. 消息页

### 页面目标

让消息中心真正承担提醒和留痕作用，而不是简单信息堆叠。

### 页面结构

#### 5.1 分组方式

消息分为两组：

- `重要通知`
- `其他消息`

#### 5.2 重要通知

包含：

- 提现状态更新
- 工单回复
- 任务即将过期
- 购买失败
- 任务完成到账

#### 5.3 其他消息

包含：

- 签到奖励
- 碎片掉落
- 推广进度
- 系统公告

#### 5.4 消息卡片字段

展示：

- 标题
- 简短摘要
- 时间
- 未读状态

操作：

- 点击进入
- 一键全部已读

要求：

- 顶部浮动通知只用于高优先级消息
- 普通通知留在消息页内查看

## 6. 我的页

### 页面目标

让“我的”页成为用户的账户总控台，而不是纯菜单页。

### 页面结构

#### 6.1 顶部账户卡

展示：

- 默认头像
- 脱敏账号 ID
- 手机号
- 推广码
- 注册时间

按钮：

- `复制推广码`

#### 6.2 余额摘要区

展示：

- 系统余额
- 任务余额
- 当前提现状态

建议文案：

- `系统余额：可用于购买和提现`
- `任务余额：需先划转后提现`
- `当前状态：已满足提现门槛 / 还差 {金额}`

要求：

- 首屏必须看到余额
- 不允许把余额埋到钱包页二级入口后面

#### 6.3 常用服务区

优先展示：

- 钱包中心
- 订单列表
- 联系客服
- 消息通知

#### 6.4 其他服务区

再展示：

- 提现排行榜
- 碎片背包
- 绑定 WhatsApp
- 密码与找回

原则：

- 高频服务在前
- 设置类入口后置

## 7. 钱包页

### 页面目标

让用户理解钱的流转规则和提现条件。

### 页面结构

#### 7.1 顶部余额区

展示：

- 系统余额
- 任务余额
- 提现门槛
- 当前可提现状态

#### 7.2 规则说明区

固定文案建议：

- `充值金额将进入系统余额`
- `购买商品将扣除系统余额`
- `任务完成后的奖励将进入任务余额`
- `任务余额需先划转到系统余额后才能提现`
- `未达到提现门槛时暂不可提现`

要求：

- 每条规则单独展示
- 不写成长段说明文

#### 7.3 资金动作区

动作卡：

- 充值
- 划转
- 提现

每个动作附一句解释：

- `充值：用于补充系统余额`
- `划转：将任务余额转入系统余额`
- `提现：从系统余额发起提现`

#### 7.4 提现状态区

展示状态及对应中文文案：

- `submitted` -> `已提交`
- `reviewing` -> `审核中`
- `approved` -> `已通过`
- `rejected` -> `已拒绝`
- `paid` -> `已打款`

#### 7.5 历史资金明细

字段：

- 类型
- 金额
- 状态
- 时间

建议加简单筛选：

- 全部
- 充值
- 扣款
- 奖励
- 划转
- 提现

### 页面结构

#### 8.1 顶部筛选

- 全部
- 成功
- 失败
- 处理中

#### 8.2 列表字段

- 订单号
- 金额
- 状态
- 创建时间
- 来源类型

来源类型文案：

- `任务包订单`

- 首版不做订单详情页

字段：

#### 9.4 工单详情页

## 11. 碎片页

### 页面目标

作为留存玩法存在，不抢主任务链路的注意力。

### 页面结构

展示：

- 当前拥有的碎片
- 合成进度
- 掉落记录
- 兑换入口
- 邮寄状态

原则：

- 强化收集感
- 弱化首页占位

## 12. 统一文案规范

所有资金与失败场景统一遵循以下顺序：

1. 先说结果
2. 再说原因
3. 最后给动作

示例：

- `提现未通过`
- `当前系统余额未达到提现门槛`
- `去充值`

或：

- `任务包已失效`
- `该任务包已超过规定完成时间`
- `返回任务中心`

原则：

- 少术语
- 多解释
- 一定要有下一步

---

## 13. 给设计与前端的执行指令

### 给 UI / UX 设计

- 首页首屏只能有一个主动作
- 钱包与“我的”页必须强化余额可见性
- 任务卡片主次信息分层明确
- 高优先级消息与普通消息视觉分层
- 资金与状态页面优先清晰，不优先花哨

### 给前端实现

- 所有失败状态必须挂动作按钮
- 提示浮层不下压页面
- 倒计时、进度、余额状态需稳定可读
- 所有组件预留英文文案长度
- 按钮长度不能因英文扩展导致挤压错位

---

## 14. 本稿结论

v6 的目标不是“功能更丰富”，而是“主链路更顺、规则更懂、资金更可信”。

后续如果进入设计或开发阶段，应优先按以下顺序执行：

1. 首页主动作整改
2. “我的”页首屏余额化
3. 钱包说明重构
4. 领取确认弹层
5. 任务失败补救动作
6. 消息优先级分层
